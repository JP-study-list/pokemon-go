# IV100 圖鑑

Pokémon GO 傳說寶可夢收集進度追蹤工具。記錄 XXL、XXS、IV100（含異色 / 亮晶晶）與活動背卡，
支援繁中・日文・英文切換，用 Google 帳號登入後自動跨裝置同步。

---

## 檔案結構

```
index.html            頁面骨架（頂部列 + 側欄 + 主內容容器）
css/style.css         全部樣式，設計 token 集中在 :root
js/config.js          Firebase 設定
js/data.js            68 隻寶可夢的基礎資料
js/i18n.js            三語字典
js/types.js           屬性顏色與三語名稱
js/backgrounds.js     活動背卡資料
js/pokedex.js         全國圖鑑名稱對照（背卡用，1025 隻）
js/pikachu.js         裝扮皮卡丘（95 種）
js/moves.js           團體戰最佳配招
js/share.js           交換清單分享圖（canvas）
img/bg/               背卡圖片
js/auth.js            Google 登入
js/store.js           Firestore 讀寫
js/ui.js              畫面繪製
js/main.js            進入點，把各模組接起來
firestore.rules       安全規則（需手動貼到 Console）
```

拆檔的用意是**改一件事只碰一個檔案**：

| 想做的事 | 改哪裡 |
| --- | --- |
| 新增寶可夢 | `js/data.js` |
| 改配色、字級、間距 | `css/style.css` 的 `:root` |
| 改介面文字、加語言 | `js/i18n.js` |
| 改屬性配色 | `js/types.js` |
| 修改配招資料 | `js/moves.js` |
| 新增活動背卡 | `js/backgrounds.js` + `img/bg/` |
| 新增裝扮皮卡丘 | `js/pikachu.js` |
| 換 Firebase 專案 | `js/config.js` |
| 改卡片或詳情面板長相 | `js/ui.js` |
| 改側欄寬度 | `css/style.css` 的 `--side-w` |

## 首次設定

### 1. Firebase Console

**Authentication**
1. 建構 → Authentication → 開始使用
2. Sign-in method → Google → 啟用 → 選支援電子郵件 → 儲存
3. 同一頁的「已授權網域」→ 新增 `jp-study-list.github.io`
   （`localhost` 預設已存在，本機測試不用另外加）

**Firestore**
1. 建構 → Firestore Database → 建立資料庫
2. 位置選 `asia-east1`
3. 建好後進「規則」分頁，把 `firestore.rules` 的內容整段貼上 → 發布

> 沒完成第 3 步的話，所有讀寫都會被拒絕，程式會顯示「載入失敗」。

### 2. 部署到 GitHub Pages

1. 建 repo `pokemon-go`
2. 用 GitHub Desktop 把整個資料夾 commit + push
3. Settings → Pages → Source 選 `main` / `/ (root)` → Save
4. 等一兩分鐘，網址是 `https://jp-study-list.github.io/pokemon-go/`

## 新增寶可夢

在 `js/data.js` 的陣列加一行：

```js
{ id:"p068", no:888, art:888, gen:8, zh:"蒼響", ja:"ザシアン", en:"Zacian", cp20:null, cp25:null, types:["fairy"] },
```

規則：

- **`id` 一旦發布就不要改。** 它是使用者紀錄的對應鍵，改了會讓既有紀錄對不上。
  新增時用還沒用過的號碼往下接即可。
- `art` 是官方立繪的編號，作為備援。一般等於 `no`；不同型態要用 PokeAPI
  的專屬編號，可在 `https://pokeapi.co/api/v2/pokemon/{名稱}` 查到。
- `go` 是 Pokémon GO 遊戲內圖示的檔名，優先使用。
  格式為 `pm{編號}.icon.png`，有型態變化的要加後綴（例如 `pm487.fORIGIN.icon.png`）。
  來源是 PokeMiners 的 `Images/Pokemon/Addressable Assets/`。
- `cp20` / `cp25` 填**IV 100% 時的捕捉 CP**（LV20 是一般團戰，LV25 是天氣加成）。
  不知道就填 `null`，畫面會顯示「—」。
- `types` 用屬性代號，可用的代號見 `js/types.js`。單屬性就寫一個。

刪除寶可夢時只要把那行拿掉，使用者資料庫裡的殘留紀錄會被自動忽略，不用另外清理。

## 寶可夢圖片

優先顯示 Pokémon GO 的遊戲內圖示，載入失敗時自動退回官方立繪。

傳說 79 隻的 GO 圖示全部驗證可用；背卡裡的一般寶可夢覆蓋率約 86%，
缺的多半是有型態變化或第 9 代尚未實裝者，會自動走備援不會破圖
（目前只有捷拉奧拉會用到備援）。

GO 圖示不是正方形，尺寸也各不相同，所以畫面一律用 `object-fit: contain`
與等比縮放，不會變形。

## 資料設計

Firestore **只存收集狀態**，不存名稱和 CP：

```
users/{uid}
  states: { p000: { xxl, xxs, iv100: { has, shiny, lucky } }, ... }
  updated: 時間戳
```

名稱、圖鑑編號、CP 一律以 `data.js` 為準。這樣之後修正名稱或新增寶可夢，
所有使用者的既有紀錄都不受影響，不需要做資料遷移。

另外，狀態全部是預設值的寶可夢不會被寫入，所以文件通常很小。

## 安全性

- `js/config.js` 裡的 `apiKey` 出現在前端是正常的，那不是密碼
- 真正的防護在 `firestore.rules`：每個人只能存取自己 `uid` 底下的文件
- 沒有登入就完全無法讀寫任何資料

**注意**：規則裡限制了單一文件最多 2000 筆狀態，避免有人把你的專案當免費儲存空間。
如果之後寶可夢數量超過這個值，記得同步調整。

## 背卡圖鑑

側欄「檢視」可切換寶可夢圖鑑與背卡圖鑑。背卡圖鑑分兩層：
背卡大圖網格 → 該背卡的寶可夢方格（點一下切換有／沒有）。

新增一個活動：在 `js/backgrounds.js` 的 `EVENTS` 加一筆，
圖片放進 `img/bg/`，檔名要跟 `card.img` 一致。

背卡的 `pokemon` 陣列有兩種寫法：

```js
"p003"                                    // 圖鑑（data.js）裡的傳說
{ dex: 131 }                              // 一般寶可夢，用全國圖鑑編號
{ dex: 131, note_zh: "布蘭琪風", ... }     // 特殊造型，合併本體但加註
```

造型一律合併成本體（帕底亞肯泰羅三種都算「肯泰羅」一格），
要區分時用 `note_zh` / `note_ja` / `note_en` 註記，會顯示在名稱下方。
一般寶可夢的名稱和圖片自動查 `js/pokedex.js`，不需另外維護。

### 異色亮晶晶

「異色亮晶晶」是獨立的收藏成就，不由異色與亮晶晶推導。
開啟它會自動帶上異色與亮晶晶，並鎖住這兩個開關，
必須先取消異色亮晶晶才能個別修改，避免誤刪紀錄。

傳說寶可夢（IV100 底下）與裝扮皮卡丘都適用同一套規則。

裝扮皮卡丘的儲存格式在鍵尾加後綴，可疊加：

```
reds-hat     一般
reds-hat*    異色
reds-hat+    亮晶晶
reds-hat*+#  異色亮晶晶
```

舊資料只有 `*`，讀進來行為不變，不需要轉換。

### 異色背卡

每個背卡格子左下角有異色標記（✦）。異色是背卡的子狀態：
必須先有該背卡才能標記異色，取消背卡時異色會一併清除。

儲存格式是在項目鍵後面加 `*`，例如 `"p003*"`、`"d131*"`。
舊資料沒有 `*`，讀進來就是非異色，不需要轉換。

背卡的異色與主圖鑑 IV100 底下的「異色」是兩件獨立的事：
前者是「這張背卡的那隻是異色」，後者是「我有一隻異色且 IV100 的」。

### 命名規則

活動 id：活動類型 + 年份，有子分類加後綴（`gofest2026`、`gofest2026-inperson`）
背卡 id：活動縮寫 + 年份後兩碼 + 識別名（`gf26-tokyo`、`gf26-global`）
圖片檔名：完整活動名，比 id 好讀（`gofest2026-tokyo.jpg`）

**背卡 id 一旦發布就不能改**，它是使用者紀錄的鍵，跟 `data.js` 的 `p###` 同樣道理。

`scope` 分兩種：`global`（全球活動）與 `regional`（地區限定，顯示為粉紅標籤）。

## 交換清單

側欄「檢視」的第四項。共有**四份清單**，可以分別給不同的人看，
每份都能重新命名（例如「跟小明換」）。上方分頁切換，數字是該份的項目數。

在寶可夢圖鑑點開任一隻，面板下方「新增需求」會加到目前這份清單。

**同一隻可以有多筆需求**，因為「異色超夢」和「有東京背卡的超夢」
是兩個獨立的交換目標。每筆可分別指定 XXL、XXS、異色，以及想要哪一張背卡。

新增時預設勾選異色。清單頁上方有「加入所有缺 IV100」，
把還沒 IV100 的傳說全部加進目前這份清單——交換的價值就在於 IV 重骰，
所以想換的通常正是這些。重複執行不會產生重複項目。

「產生分享圖」只輸出目前這份清單，圖片標題會用清單名稱。

清單頁的方格會把背卡疊在立繪後方（跟遊戲裡一樣），
左上角標出 XXL / XXS / 異色。

「產生分享圖」會用 canvas 畫出一張 PNG 直接下載，可以傳給朋友。
圖片為兩倍解析度，深色模式下會輸出深色版本。

只有傳說寶可夢有這個功能。清單不記錄 IV100，
因為交換後 IV 會重骰——想交換的本來就是還沒 IV100 的那些。

儲存格式是 `{ lists: [{name, items}, ...] }`。
先前的兩種舊格式（單一陣列、一隻一筆的物件）都會在讀取時自動轉成第一份清單，
不需要手動遷移。

## 團體戰配招

傳說寶可夢的詳情面板會顯示攻擊向的前兩名配招，招式標上屬性顏色，
★ 表示傳授招式（需要招式學習器）。

資料取自 PokeMiners 的官方 game master，用循環 DPS 公式計算排名：

```
n     = 特殊招式耗能 / 一般招式集能
DPS   = (n × 一般威力 + 特殊威力) / (n × 一般時間 + 特殊時間)
```

屬性一致時威力乘 1.2。採中性對戰，不考慮特定王的相剋與閃避。
招式的三語名稱取自遊戲內官方翻譯檔。

Niantic 調整招式數值後排名可能改變，屆時需重新產生 `js/moves.js`。

## 皮卡丘圖鑑

側欄「檢視」的第三項。95 種裝扮皮卡丘的方格牆，
點格子開啟詳情面板，裡面有「裝扮」開關，以及三個子狀態：
異色、亮晶晶、異色亮晶晶。
若該裝扮出現在某些背卡上，面板下方會列出來，可直接勾選——
這些紀錄與背卡圖鑑共用同一筆資料，兩邊即時同步。

圖片來自 `Choggor/Pikachu-costume-tracker`，不需要自己存檔。
中日文名稱是對照翻譯，可能與遊戲內用詞不同，發現有誤直接改 `js/pikachu.js`。

背卡資料可以指定裝扮，格子就會顯示對應的帽子圖：

```js
{ dex: 25, costume: "gotour-2026-calems-hat", note_zh: "卡爾姆帽" }
```

鍵會變成 `d25-gotour-2026-calems-hat`，所以同一張背卡可以有多隻不同裝扮的皮卡丘。

## 版面

- **頂部列**：漢堡鈕、置中標題、深色模式、搜尋框
- **側欄**（264px）：檢視、收集統計、篩選，底部是設定齒輪與頭像
- **設定選單**：點齒輪展開，內含語言切換與登出
- 側欄未來要加功能，就在 `index.html` 的 `.side-scroll` 裡加一個 `<section class="side-block">`
- 桌機會記住上次的收合狀態；手機預設收合，選完篩選自動關閉
